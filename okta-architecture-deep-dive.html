Section 01

IdP（Identity Provider）の基礎

IdPとは「誰であるか」を証明するサービス。ユーザーが何かにアクセスしようとしたとき、IdPが認証を担い、結果を信頼するSP（Service Provider）にアサーションを渡す。

IdP（Identity Provider）認証側

ユーザーのクレデンシャルを管理・検証し、認証トークン/アサーションを発行する。例：Okta、Entra ID、Google Workspace

SP（Service Provider）リソース側

IdPを信頼してアクセスを許可するサービス。例：Salesforce、M365、社内アプリ。認証ロジックを持たず、IdPのアサーションを検証するだけ。

認証フローの基本

ユーザー
ブラウザ
① アクセス要求
SP
Salesforce等
② IdPへリダイレクト
IdP（Okta）
認証・MFA
③ アサーション発行
SP
アクセス許可
SPは「Oktaが認証した」という事実だけを信頼する。パスワードはSPに届かない。これがSSOの本質。

フェデレーション（Identity Federation）とは

異なる組織・ドメイン・システム間でIdPの信頼を連携させる仕組み。「AのIdPを信頼する」という設定をBのSPが持つことで、Aで認証したユーザーがBのサービスをシームレスに使える。

フェデレーション設定例：Entra ID ⇄ Okta
# Entra ID側でのフェデレーション設定概念 Entra ID > External Identities > External IdP登録 → Type: SAML/WS-Fed → Metadata URL: https://<oktadomain>/app/<appid>/sso/saml/metadata → Federated Domain: contoso.com を Okta に委譲 # 結果 user@contoso.com でM365ログイン → Entra IDが「このドメインはOktaで認証」と判断 → Oktaにリダイレクト → Okta MFA通過 → SAMLアサーションをEntra IDに返却 → M365アクセス許可
Section 02

認証・認可プロトコル

SAML 2.0
Security Assertion Markup Language
XMLベースのフェデレーションプロトコル。エンタープライズSSOの主流。IdPとSPがXMLアサーションを交換する。2005年策定で古いが普及率が高い。
IdP-Initiated SSO SP-Initiated SSO
OIDC
OpenID Connect
OAuth 2.0の上に認証レイヤーを追加したプロトコル。JSONベース。モダンアプリ・SPAに適する。IDトークン（JWT）を発行。Oktaが特に推進。
JWT モダンアプリ向け
OAuth 2.0
Open Authorization
認可のフレームワーク。「誰か」を証明するのではなく「何にアクセスできるか」を委譲する。アクセストークンを発行。OIDCと組み合わせて使うことが多い。
アクセス委譲 APIアクセス制御
SCIM 2.0
System for Cross-domain Identity Management
ユーザープロビジョニング自動化の標準プロトコル。IdPからSPへのアカウント作成/更新/削除をAPI経由で自動化。Okta Lifecycle Managementの根幹。
プロビジョニング
LDAP
Lightweight Directory Access Protocol
Active DirectoryやOpenLDAPとの連携に使用。Okta AD Agent/LDAP Interfaceを介してオンプレADとの同期・認証を実現する。
レガシー連携
RADIUS
Remote Authentication Dial-In User Service
ネットワーク機器（VPN/Wi-Fi/スイッチ）の認証に使用。Okta RADIUS Agentを挟むことでOktaのMFAをネットワーク機器認証に適用できる。
NW機器認証

SAMLとOIDCの使い分け

観点 SAML 2.0 OIDC / OAuth 2.0
フォーマット XML（重い） JSON / JWT（軽い）
向いてる用途 エンタープライズSaaS、レガシー統合 モダンアプリ、SPA、モバイル、API
主なSP例 Salesforce、ServiceNow、旧来SaaS 社内アプリ、Slack、GitHub、Google
設定複雑度 中〜高（メタデータ交換） 低（client_id / secret）
トークン SAMLアサーション（XML署名） ID Token + Access Token（JWT）
Section 03

Okta 製品ポートフォリオ

Oktaは大きく「Workforce Identity Cloud」と「Customer Identity Cloud（旧Auth0）」の2系統で構成される。

Workforce Identity Cloud（WIC）— 社員向け

SSO / Universal Directory

全アプリへの統一ログイン。7,000+のOINアプリ統合。ユーザーディレクトリの一元管理（AD/HR/LDAP同期）

Adaptive MFA

リスクベースの多要素認証。ロケーション/デバイス/行動分析でステップアップ認証を動的に制御

Lifecycle Management

SCIM + Workflowsによるプロビジョニング自動化。入社/異動/退社トリガーでSaaS全体のアカウントを自動制御

Identity Governance（OIG）

アクセスレビュー、権限認定、SOD（職務分離）。Saviyntほど重厚ではないが軽量IGAとして機能

Privileged Access（OPA）

サーバー/インフラへのJIT（Just-in-Time）アクセス。SSH/RDPセッション制御。CyberArkの軽量代替

Device Access

デバイス証明書によるパスワードレス認証。macOS/Windows対応。Entra IDのHybrid Joinに相当する機能

Okta Workflows

ノーコードの自動化エンジン。HR連携、Slack通知、チケット起票、承認フローなどをコードなしで構築

Advanced Server Access（ASA）

クラウドサーバーへのゼロトラストアクセス。SSH証明書の短命化、セッション録画、アクセスログ

Okta Verify / FastPass

デバイスバインドのパスワードレス認証。生体認証連携。フィッシング耐性のある認証（FIDO2相当）

Customer Identity Cloud（CIC）— 顧客向け＝旧Auth0

B2C（消費者向けサービス）Auth0
  • ソーシャルログイン（Google/LINE/Facebook）
  • カスタマイズ可能なログインUI
  • パスワードレス（magic link/OTP）
  • MFAとリスクベース認証
  • 月間アクティブユーザー数ベース課金
B2B（企業顧客へのサービス提供）Auth0
  • マルチテナントSaaS向けのIdP
  • 顧客企業ごとのEntra ID/Okta連携
  • Organizations機能でテナント分離
  • 顧客がSSO設定を自己管理できるAdmin Portal
WIC（Workforce）とCIC（Customer）は同じOktaブランドだが、アーキテクチャもコードベースも異なる別製品。混同注意。
Section 04

実際の構成パターン

1
Okta = Primary IdP、Entra IDをフェデレーション配下に置く
最頻出

エンタープライズで最も多いパターン。M365も含む全SaaSをOktaで統一管理し、Entra IDはOktaからSAMLを受け取るだけのSPに降格させる。

社員 → Okta（Primary IdP） │ ├── SAML/OIDC → Entra ID（M365） ← フェデレーション ├── SAML → Salesforce ├── SCIM → Workday ├── SAML → ServiceNow ├── OIDC → 社内カスタムアプリ └── RADIUS → VPN / Wi-Fi
Entra ID側での設定：ドメインをFederated Domainとしてマーク → パスワードハッシュはAzure ADに同期しない（認証をOktaに完全委譲）→ Conditional Accessは引き続きEntra ID側で動く
メリット
  • ユーザーがOktaだけ覚えればよい
  • 全SaaSの認証ポリシーを一元管理
  • 非Microsoft SaaSの統合品質が高い
  • Lifecycle Managementで全アプリ自動プロビジョニング
制約・注意点
  • Entra ID Conditional Accessは残る（二重管理）
  • Windows Hello / Autopilotはまだ深くEntra依存
  • Okta障害時はM365も含めて全サービスが影響を受ける
2
Entra ID = Primary IdP（MicrosoftスタックのみでOktaなし）
Microsoft中心組織

M365 E5 + Entra P2を持ち、SaaSがほぼMicrosoft系の場合。コストとシンプルさが最大のメリット。

社員 → Entra ID（Primary IdP） │ ├── Native → M365 / Azure ├── SAML → Salesforce（サポートあるが設定はOktaより手動） ├── SCIM → 一部SaaS（サポート範囲は限定的） └── AD Connect → オンプレAD同期
注意：Entra IDのSCIMプロビジョニングは非Microsoft SaaSへの対応が薄い。Workday連携もOktaより設定が複雑。SaaS増加フェーズで管理コスト急増のリスクあり。
3
Okta WIC（従業員）+ Auth0 CIC（顧客）の二層構成
B2C SaaS企業

自社サービスを外部顧客に提供しながら、社内の従業員もOktaで管理。Okta製品で全IDを完結させる戦略。

[従業員] 社員 → Okta WIC → 社内SaaS全般 [外部顧客] 顧客 → Auth0（CIC） ├── ソーシャルログイン（Google/LINE） ├── パスワードレス（Magic Link） └── エンタープライズ顧客 → 顧客のEntra IDやOktaにフェデレーション
4
AD → Okta → 全SaaS（AD移行期の典型）
AD現役組織のモダナイゼーション

オンプレADが残っており、Entra IDへの完全移行が難しい状況。Okta AD AgentでADとOktaを同期させ、クラウドSaaSはOktaで管理。

オンプレAD ←→ Okta AD Agent（双方向同期） ↓ Okta（IdP） │ ├── SAML/OIDC → 全クラウドSaaS ├── フェデレーション → Entra ID（M365） └── RADIUS → VPN # AD Agentは軽量なサービスとしてADドメインコントローラー近くに配置 # Oktaのパスワード変更をADに書き戻すことも可能
AD移行の中間ステップとして非常に有効。ADを捨てずにクラウドSaaSを統合できる。最終的にはEntra ID Hybrid Joinと並行運用するケースが多い。
Section 05

Okta 主要コンポーネント詳細

Universal Directory（UD）— マスターIDストア

Oktaのユーザーディレクトリ。AD・HR・LDAP・SaaSからユーザー情報を集約し、単一の「信頼できるプロファイル」を維持する。

AD（オンプレ） ──→ Workday（HR） ──→ Universal Directory ──→ 全SaaSへSCIM配信 Google Workspace ──→ （マスター） カスタムアプリ ──→
  • カスタム属性の追加が自由（部署コード、コスト番号、etc）
  • 属性マッピングルールで各SaaSに適切な値を送信
  • プロファイル変更の優先ソース設定（どのシステムが正）

Adaptive MFA — リスクベース認証

認証要素 種類 フィッシング耐性
Okta Verify (Push) プッシュ通知承認 △（MFA疲労攻撃に弱い）
Okta FastPass デバイスバインド生体認証 ◎（フィッシング耐性）
FIDO2 / WebAuthn ハードウェアキー ◎（最強）
TOTP（Google Authenticator等） 時刻ベースOTP ✕（AiTMに脆弱）
SMS OTP テキストメッセージ ✕（SIMスワップ）

Adaptive = ユーザーのリスクスコアに応じてMFAのステップを動的に変える。通常の場所・デバイスからは要求しない、未知の環境からはFastPass必須、など。

Lifecycle Management — 自動プロビジョニング

1
Joiner（入社）

WorkdayにHRレコードが作成 → OktaがSCIM/Webhookで受信 → 全SaaS（M365/Salesforce/Slack等）のアカウントを自動作成、グループ付与

2
Mover（異動）

部署変更 → Okta Workflowsが既存グループから外し新グループに追加 → SaaS権限が自動更新（例：営業ツールのアクセス削除、新部門ツール付与）

3
Leaver（退社）

HRで退社ステータス変更 → Okta全アカウント即時無効化 → 設定日数後に全SaaSアカウント削除 → M365ライセンス回収

Entra IDのプロビジョニングと比較して、非Microsoft SaaSへのSCIM対応数（Okta OIN: 7,000+）が圧倒的に多い。Workday→Salesforce等の組み合わせでEntra IDより設定が楽。

Okta Workflows（ノーコード自動化）

できること
  • HR連携トリガーでSlackチャンネル自動作成
  • 特権グループ参加時にServiceNowチケット起票
  • 退社時にGoogleドライブデータをオーナー移転
  • アクセスレビュー未回答者への自動リマインダー
  • 外部API呼び出し（REST/GraphQL）
競合との位置づけ

Power Automate（Microsoft）に近い概念だが、Identityイベント（ユーザー作成/グループ変更/認証失敗）をネイティブトリガーにできる点が差別化。Saviyntのライフサイクルエンジンより軽量。

Section 07

選定基準 — どの状況でOktaを推すか

Okta が刺さる状況
  • SaaSが多様で非Microsoft（Salesforce/Workday/ServiceNow中心）
  • AWS/GCPとAzureを併用するマルチクラウド環境
  • B2C・B2B問わず外部ユーザーのID管理が必要（Auth0）
  • 入退社プロセスの自動化が課題でHRシステムがある
  • Microsoft依存リスクを避けたい（金融・インフラ系）
  • M&Aで複数テナントの統合が発生する
  • グローバル展開でテナント管理が複雑
Entra IDだけで充分な状況
  • M365 + Azure中心で非Microsoft SaaSがほぼない
  • E3/E5ライセンスを既に全社展開している
  • WindowsデバイスとAutopilotが主要な管理対象
  • 組織規模が小さく、SaaSの数が限定的（20本以下）
  • IT予算が限られてOkta追加投資が難しい

ヒアリング時の確認ポイント

質問 Okta推しになる回答
今使っているSaaS上位10本は？ Salesforce/Workday/ServiceNow/Slack等の非Microsoft SaaSが複数
入退社の手続きはどのくらい時間かかる？ 「数日かかる」「手動でIT部門が作業」「退社時の漏れが心配」
クラウドはAzureだけ？ 「AWS/GCPも使っている」「マルチクラウドを推進したい」
外部パートナー・顧客向けのID管理は？ 「外部ユーザーが多い」「顧客に自社サービスを提供している」
MicrosoftのE5を全員分持っている？ 「一部だけ」「E3止まり」「Entra P2は別途買っていない」
Presalesポイント：「Entra IDがあるからOktaは要らない」という顧客には、現在のSaaS棚卸し（何本使っているか、プロビジョニングが自動化されているか）を先に一緒に確認するのが最速の突破口。
