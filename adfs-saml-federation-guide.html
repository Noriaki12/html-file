🤔SSOとは何か
SSO シングルサインオン（Single Sign-On）

一度ログインするだけで、複数のサービスを追加認証なしで使えるようにする仕組み。
ユーザーが「会社のPCにログイン済み」→「給与計算SaaS、Slack、Salesforceも自動でログイン」という状態。

🎡 わかりやすいたとえ話 遊園地の「1日フリーパス」。入場口で一度だけリストバンドをもらえば、園内のどのアトラクションも都度チケットを買わずに乗れる。 SSOの「認証トークン」= フリーパスのリストバンド。
問題設定 なぜSSOが必要か

SSO がない場合（= 今回の問題の初期状態）：

社員AD に Windows ログイン済み
→
クラウド給与SaaS「パスワードを入力してください」😩
😤 課題 サービスごとにID/パスワードが必要 → 利便性低下 + パスワード使い回しリスク + ヘルプデスクコスト増

🖥️AD FS とは
AD FS Active Directory Federation Services

オンプレミスに置くフェデレーション認証サーバー。
AD に保存されているユーザー情報をもとに、信頼した外部サービスへのSSOを可能にするトークン発行機関。

🔏 AD FS のイメージ 銀行が発行する「本人確認済み証明書」を思い浮かべてほしい。
銀行（= AD FS）が「この人は確かに口座保有者です（本人確認済み）」という証明書（= SAMLトークン）を発行。
提携店（= 給与計算SaaS）はその証明書を見て「銀行が保証するなら信頼する」と判断してログインを通す。
AD FS の核心：IdP（Identity Provider）という役割
役割名略称説明今回の問題では
Identity Provider IdP 「このユーザーは誰か」を証明するサービス AD FS（社内サーバー）
Service Provider SP IdPの証明を信頼してサービスを提供する側 給与計算クラウドSaaS
💡 ポイント AD FS は IdP として「トークン（証明書）」を発行する。給与計算SaaSはSPとしてそのトークンを受け取りログインを許可する。この「信頼関係の設定」が今回の問題の核心。
AD FS と Entra ID（旧Azure AD）はどう違う？
比較項目AD FSEntra ID（Azure AD）
設置場所 オンプレミス（自社サーバー） Microsoft クラウド
管理コスト 高い（自社で運用・保守） 低い（Microsoftが管理）
認証プロトコル 主に SAML / WS-Federation SAML / OIDC / OAuth 2.0
今回の問題での位置 メインの IdP（認証基盤） Entra Connect で同期先として使用
📄SAML とは
SAML Security Assertion Markup Language

IdP（AD FS）と SP（クラウドSaaS）の間でユーザー認証情報を安全にやりとりするためのXMLベースの標準規格。
「このユーザーが認証済みであることを、どんな形式で・どんな情報を含めて伝えるか」を定めたルール。

📮 SAMLのたとえ話 ビザ申請を想像してほしい。
日本の外務省（= AD FS）が「この人は日本国民で、パスポート番号はXXXXです」と書いた公式書類（= SAMLアサーション）を発行。
渡航先の入国審査（= 給与計算SaaS）はその書類を見て入国（= ログイン）を許可する。
SAMLは「その書類の書き方の国際標準」。
SAMLに登場する3つのデータ
A
SAMLアサーション（Assertion）

「ユーザーが認証済みであること」を証明するXMLデータ本体。ユーザー名・メール・グループ情報などを含む。AD FSが発行し、デジタル署名が付く。

B
SAMLリクエスト（Request）

SP（給与計算SaaS）が「認証してください」とIdP（AD FS）に送るリクエスト。

C
SAMLレスポンス（Response）

IdP（AD FS）が「認証OKです。このアサーションを持って帰ってください」とSPに返す応答。アサーションを含む。

重要 SAMLメタデータとは何か

SAMLメタデータは「IdPとSPが互いの設定情報を交換するためのXMLファイル」。
信頼関係を構築するために相手に渡す設定書類。

entityID"https://adfs.company.com/adfs/services/trust"— AD FSの識別子
SingleSignOnService"https://adfs.company.com/adfs/ls/"— SSOのエンドポイントURL
SingleLogoutService"https://adfs.company.com/adfs/ls/?wa=wsignout1.0"— ログアウトURL
X509Certificate"MIIDXTCCAkWg..." — トークン署名証明書（公開鍵）
NameIDFormat"emailAddress"— ユーザーの識別方式
⚠️ 「証明書だけ送る」より「メタデータを送る」方が上位互換 選択肢①の「AD FSのトークン署名証明書を送信」は証明書（上記のX509Certificate）のみを渡す行為。 選択肢②の「SAMLメタデータを送信」はエンドポイントURL・証明書・識別子などすべてを含む。試験では②が正解になる理由がこれ。
🔄SSO認証フロー（全体の流れ）
フェデレーション SSO が動く仕組み（SP-Initiated）

ユーザーがクラウドSaaS（SP）にアクセスしてSSOが発動するパターン。最も一般的。

1
ユーザーが給与計算SaaSにアクセス

ブラウザで payroll.example.com を開く。
SaaSはログインしていないユーザーを検出。

2
SP → IdP へリダイレクト（SAMLリクエスト）

SaaSが「認証してください」とAD FS（社内サーバー）へSAMLリクエストを含むリダイレクトを返す。
ユーザーは自動的に adfs.company.com へ飛ばされる。

3
AD FS がユーザーを認証

AD FSはWindows統合認証（Kerberos/NTLM）でドメイン参加PCのユーザーをプロンプトなしで認証。
AD に照合し、認証OK なら SAMLアサーションを生成。

4
AD FS → SP へ SAMLアサーション送信

署名付きのSAMLアサーション（「このユーザーは◯◯で認証済み」）をブラウザ経由でSaaSへ送る。

5
SP がアサーションを検証してログイン完了

SaaSはAD FSの公開鍵（メタデータ内の証明書）でアサーションの署名を検証。
問題なければログイン完了。ユーザーにはパスワード入力画面は一切表示されない。

⚙️フェデレーション設定に必要な2つの操作
🔑 核心の考え方 フェデレーションは「双方向の信頼設定」。自社AD FS側の設定（③）と、相手SaaS側への情報提供（②）の両方が必要。片方だけでは動かない。
操作 ③ Relying Party Trust（証明書利用者の信頼）を追加

自社AD FSコンソール上で「この給与計算SaaSにはトークンを発行してよい」と登録する操作。

1
AD FSコンソールを開く

Windows Server の AD FS管理ツール（Server Manager または MMC スナップイン）を起動。

2
「証明書利用者の信頼の追加」ウィザードを実行

給与計算SaaSのSAMLメタデータURLまたはファイルを読み込む。相手のEntityID・エンドポイントURLが登録される。

3
クレームルールを設定

「どのユーザー属性（名前・メール・グループ等）をトークンに含めてSaaSに渡すか」を設定。

🏪 Relying Party Trust のたとえ話 ホテルのカードキーシステムの「許可リスト」。
フロント（= AD FS）が「このホテルのカードキーはジムにも使えますよ」とジム（= 給与計算SaaS）側の名前をシステムに登録する行為。
登録がなければ、どんな正規のカードキーを持っていても入れない。
操作 ② SAMLメタデータを給与計算プロバイダーに送信

相手SaaS側に「うちのAD FSはこういう設定ですよ」という情報を渡す操作。

AD FSのメタデータは以下のURLで公開されている：

メタデータURL https://[AD FSサーバー名]/FederationMetadata/2007-06/FederationMetadata.xml

このメタデータに含まれる情報：

含まれる情報給与SaaSが何に使うか
エンドポイントURL SAMLリクエストをどこに送ればよいか知るため
トークン署名証明書（公開鍵） 受け取ったアサーションの署名が本物か検証するため
EntityID どのAD FSと通信するか識別するため
NameID形式 ユーザー識別子の形式（メールアドレス等）を合わせるため
📋 メタデータ送付のたとえ 銀行（= AD FS）の「会社案内パンフレット」を提携店（= 給与計算SaaS）に渡す行為。
「うちの銀行の住所（URL）はここで、判子（証明書）はこういう形ですよ」という情報を渡すことで、
提携店が「確かに本物の銀行からの証明書だ」と確認できるようになる。
NG操作 Azure portal でアプリ登録（選択肢④）がなぜ不正解か
操作使う認証基盤今回の問題に合うか
Azure portal でアプリ登録 Entra ID（クラウドIdP） ✗ 不適合
AD FSコンソールで RPT 追加 AD FS（オンプレIdP） ✓ 適合

📝問題シナリオの完全解析

📋 問題の状況整理

  • オンプレミスの給与計算ソリューションをクラウドに移行中
  • Microsoft Entra Connect 構成済み（ADドメイン → Azure に同期）
  • ドメイン機能レベル：Windows Server 2012 R2
  • AD FS がシングルサインオンに使用中
  • ユーザーが新しいクラウド給与SaaSに接続すると資格情報の入力を求められる
🎯 要件：認証プロンプトなしで新しい給与計算クラウドに接続できるようにする
= 給与計算SaaSへのSSO（フェデレーション認証）を新たに構成する
🔍 「認証プロンプトが出る」原因 Entra Connect で ID同期はされているが、新しい給与計算SaaSとAD FSの間にまだ信頼関係が設定されていない。 AD FSはその SaaS を知らないのでトークンを発行しない → SaaS は AD FS に頼れないのでユーザーに直接パスワードを聞く。
✅各選択肢の解説
✗ 不正解 ① AD FS のトークン署名証明書を給与計算会社に送信する ▼

証明書だけではエンドポイントURL（どこにSAMLリクエストを送るか）が分からない。
给与計算SaaS側は証明書を持っていても AD FSへのアクセス先を知らないため、フェデレーションを開始できない。

② のSAMLメタデータを送れば証明書も含まれる メタデータには証明書＋エンドポイント＋EntityIDがすべて入っている。②が上位互換なのでこちらが正解になる。

※ただし古いシステムや証明書のみを要求するSaaSでは①を使うこともある。試験では「よりベストプラクティスな」選択肢を選ぶこと。

✓ 正解 ② SAMLメタデータファイルをダウンロードして給与計算プロバイダーに送信する ▼

相手SaaS側に「AD FSの設定情報一式」を渡す操作。これにより給与計算SaaSはAD FSを信頼できる IdP として登録できる。

メタデータに何が入っているか（再確認） ① SSOエンドポイントURL（SAMLリクエストの送り先）
② トークン署名証明書（受け取ったアサーションの署名検証用）
③ EntityID（どのAD FSか識別する一意のID）
④ NameIDフォーマット（ユーザー識別方法）

AD FSのメタデータURL：https://[サーバー]/FederationMetadata/2007-06/FederationMetadata.xml
ここからダウンロードしてSaaSのサポートに渡す、またはURLを直接SaaSに設定する。

✓ 正解 ③ AD FSコンソールで給与計算会社を信頼された当事者の信頼として追加する ▼

自社AD FS側に「この給与計算SaaSはトークン発行先として信頼する」と登録する操作。 英語では "Relying Party Trust (RPT)" を追加する、とも言う。

なぜこれが必要か AD FSはデフォルトでは誰にでもSAMLトークンを発行しない。「許可リスト」にSaaSを追加しないと、ユーザーが認証しようとしてもAD FSが「知らない相手には渡せない」と拒否する。

設定内容：SaaSのEntityID・ACSエンドポイントURL・クレームルール（渡す属性の定義）など。
SaaSのSAMLメタデータURLを読み込むと自動設定される場合が多い。

✗ 不正解 ④ Azure portal 内で新しいアプリ登録を作成する ▼

「Azureポータルでアプリ登録」= Entra ID（クラウドIdP）を認証基盤として使う場合の操作。

今回はAD FSがIdP 問題文に「AD FSはシングルサインオンに使用されています」と明記。つまり認証の中心はオンプレAD FS。
Entra ID をIdPとして使うのであればこの選択肢だが、それは問題の構成と異なる。

Entra Connect はあくまで「IDデータを同期するツール」であり、認証基盤を Entra ID に切り替えるものではない。
💡正解の組み合わせと理由の一言まとめ
正解① AD FSコンソールで給与計算会社をRPT（証明書利用者の信頼）として追加
▸ 自社AD FS側の設定。「この相手にはトークンを発行してよい」とAD FSに教える
正解② SAMLメタデータをダウンロードして給与計算プロバイダーに送信
▸ 相手SaaS側への設定情報提供。「うちのAD FSはここにある。証明書はこれ」と相手に教える
🤝 双方向の信頼確立が原則 自社AD FS：「給与計算SaaSを信頼リストに追加」（③）
給与計算SaaS：「このAD FSからのトークンを信頼する設定」（② メタデータ受取後に設定）
どちらか一方だけでは SSO は機能しない。
🎯類似問題で使えるパターン認識

📌 パターン①：AD FSで新SaaSにSSO追加

「AD FS運用中に新しいクラウドSaaSへのSSO設定」→ 必ずこのセット：
RPT（証明書利用者の信頼）を追加 SAMLメタデータをSaaSに送付

📌 パターン②：IdPを見極める

問題文に「AD FSを使用中」→ IdPはAD FS（オンプレ）
「Entra IDを使用中」→ IdPはEntra ID（クラウド）
IdPが違うと設定方法・使うツールが変わる。

📌 パターン③：メタデータ vs 証明書のみ

「トークン署名証明書を送る」vs「SAMLメタデータを送る」が両方選択肢にある場合はメタデータが正解。証明書単体より情報が多い（上位互換）。

📌 パターン④：Azureアプリ登録の出番

「Azureポータルでアプリ登録」が正解になるのはEntra IDをIdPとして使う場合のみ。AD FSが主役の問題では誤り選択肢として機能する。

📌 パターン⑤：Entra Connectの役割に注意

Entra Connectは「IDデータの同期ツール」。認証基盤をクラウドに移行するわけではない。AD FSを使い続けながらEntra Connectで同期することは共存できる。

📌 パターン⑥：Seamless SSO との違い

「Entra Seamless SSO」はEntra Connect + Kerberos で実現するSSO（AD FS不要）。AD FSありの問題と混同しないこと。AD FSがある = フェデレーション設定の話。

🧠まとめ：頭に入れる一言メモ
🔑
AD FS = オンプレ IdP。SAMLトークンを発行する銀行のフロント

Entra ID（クラウド）とは別物。AD FSがある環境では認証の中心はオンプレにある。

🤝
フェデレーション = 双方向の信頼設定が必須

自社AD FS側：RPT（証明書利用者の信頼）を追加。
相手SaaS側：SAMLメタデータを受け取り設定。

📋
SAMLメタデータ = エンドポイント + 証明書 + EntityID の設定書類

証明書だけ渡す選択肢は「メタデータ送付」の劣化版。証明書のみではエンドポイントが分からない。

🚫
Azureポータルのアプリ登録 = AD FS問題では不正解

これはEntra IDがIdPのときの操作。AD FSが主役の問題では誤り選択肢として登場することが多い。
