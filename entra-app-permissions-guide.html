🏢 エンタープライズアプリとは何か
Enterprise App エンタープライズアプリケーション

Entra ID（旧Azure AD）のテナントに登録された「外部サービスとの接続設定」。
「このSaaSにはこのユーザーがアクセスできる」「このSaaSにはこのAPIの権限を与える」といった設定の入れ物。

🎡 たとえ話
会社のビルの入館証システム（Entra ID）に、「〇〇SaaS社の社員証でも入れるようにする」という設定を登録する行為。
この「登録した設定情報」1件1件が「エンタープライズアプリ」。
🔑 アプリ登録
App Registration
  • アプリを「開発者」として登録する入口
  • クライアントID・シークレットを発行
  • 自社開発アプリや独自統合に使う
  • テナント内に1つ作られる「設計図」
🏪 エンタープライズアプリ
Enterprise App
  • アプリ登録から自動生成されるインスタンス
  • ユーザー・グループの割り当て管理
  • SaaSのSSO設定・権限設定の場所
  • 実際に「使う設定」を管理する場所
💡 今回の問題でのエンタープライズアプリの役割 サードパーティ製メールスキャンシステムが Entra ID テナントに接続するために使う「接続口」。 このアプリに「どんな権限を与えるか」「誰が使えるか」が設定されている。

なぜアプリに「権限」が必要か

クラウドサービスがテナントのデータにアクセスするとき、「何を見ていい・何をしていい」を明示的に定義する必要がある。これがアクセス許可（Permission）の設定。

Entra ID が「このアプリにはどんな操作を許可するか」を管理し、許可された範囲でのみAPIアクセスを認める。

⚖️ 2種類のアクセス許可

🔑 最重要：権限は2種類しかない

  • 委任されたアクセス許可（Delegated）= ユーザーの代わりに動く
  • アプリケーション権限（Application）= アプリ自身として動く（ユーザー不在）
👤 委任された
アクセス許可
Delegated Permission
  • ユーザーがサインインして使う
  • アプリは「ユーザーの権限の範囲内」でのみ動作
  • ユーザーAが持っていない権限はアプリも使えない
  • 例：「自分のメールを読む」
  • ユーザー同意 or 管理者同意が必要
📱 使われる場面 ユーザーがブラウザでサインインして使うWebアプリ、モバイルアプリなど
🤖 アプリケーション
権限
Application Permission
  • ユーザー不在でバックグラウンド動作
  • アプリ自身のIDで動作（ユーザーに依存しない）
  • テナント全体のデータにアクセス可能
  • 例：「テナント全員のメールを読む」
  • 管理者同意のみ必要（ユーザー同意は存在しない）
🤖 使われる場面 バックエンドサービス、自動化スクリプト、デーモン、今回のメールスキャンシステム
🏦 権限の違いをたとえで理解
委任（Delegated）：「代理人カード」。銀行員（アプリ）がお客様（ユーザー）の代わりに手続きするが、お客様が持っていない権限はできない。

アプリケーション（Application）：「マスターキー」。夜間の警備員（アプリ）がビル全体を見回るためのキー。特定の人間の代理ではなく、アプリ自身の権限として動作する。
よく使うMicrosoft Graph権限の例
権限名種別できること
Mail.Read Delegated サインインしたユーザーのメールを読む
Mail.Read Application テナント全ユーザーのメールを読む
User.Read Delegated サインインしたユーザーのプロフィールを読む
User.Read.All Application テナント全ユーザーのプロフィールを読む
User.ReadBasicAll Application 全ユーザーの基本情報を読む（今回の受信者検証に相当）
⚠️ 同じ名前でも種別によって影響範囲が全然違う Mail.Read の Delegated はそのユーザーのメールだけ。Application は全員のメール。 アプリケーション権限のほうが強力なため、必ず管理者が同意する必要がある。
✅ 同意（Consent）とは何か
同意 = 「このアプリにこの権限を使わせることを許可する」という行為

アプリが Entra ID のリソースにアクセスするには、誰かが「OK」を出す必要がある。この「OK」を出すことが同意（Consent）。

📱 スマホアプリで体験済みのはず
「このアプリはあなたの連絡先にアクセスしようとしています。許可しますか？」
→ これがユーザー同意のUX。企業向けでも同じ概念が使われている。
👤 ユーザー同意
User Consent
  • 個々のユーザーが自分の範囲内で許可
  • 低リスクな委任権限（自分のデータのみ）
  • サインイン時に「許可しますか？」画面が出る
  • 管理者が設定を許可している場合のみ可能
  • 非対話型アプリには存在しない
🛡️ 管理者同意
Admin Consent
  • テナント管理者（Global Admin等）が許可
  • テナント全体に対して有効
  • 高権限の委任権限・全アプリケーション権限
  • 一度管理者同意すれば全ユーザーに適用
  • Entra 管理センターで「管理者の同意の付与」ボタン

🤖 対話型 vs 非対話型アプリ
👨‍💻 対話型
Interactive / Delegated
  • ユーザーがブラウザ・アプリでサインイン
  • ログイン画面 / 同意画面が表示される
  • ユーザーのIDトークンを使って動作
  • 権限はそのユーザーの範囲内
  • 例：Webアプリ、Teams アドイン、SaaS SSO
⚙️ 非対話型
Non-interactive / Daemon
  • バックグラウンドで自動実行
  • ログイン画面・同意画面なし
  • クライアントID + シークレット or 証明書で認証
  • アプリ自身のIDとして動作（ユーザー不在）
  • 例：バッチ処理、メールスキャン、定期レポート

非対話型ではユーザーのステップが完全に存在しない。「管理者がいつ同意したか」は事前設定で決まる。

今回の問題 メールスキャンシステムが非対話型な理由
1
メールが届いたとき自動でスキャンする必要がある

ユーザーが「スキャンして」とボタンを押すわけではない。メールが来るたびに自動でテナントのディレクトリを参照して受信者アドレスを検証する。

2
特定のユーザーとしてではなく、テナント全体にアクセスする

「Aさんのメール」「Bさんのメール」と一人ひとりの委任では動けない。テナント内の全受信者を確認できる Application 権限が必要。

3
24時間365日、ユーザーのサインインに依存せず動く

バックグラウンドサービス = 非対話型 = Application権限 = 管理者同意のみ。

📝 問題シナリオの完全解析

📋 シナリオ整理

  • サードパーティ製メールスキャンシステムを O365 の前段で運用
  • SSO・高度なスキャン・レポート・隔離機能を備える
  • 非対話型の Entra エンタープライズアプリ経由でテナントに接続
  • メール送信前に送信元アドレスを検証（受信者検証機能）

🔍 問題を解くためのキー情報 「非対話型」= アプリケーション権限 = 管理者同意のみ（ユーザー同意は存在しない）
「エンタープライズアプリの権限を確認する場所」= Microsoft Entra ID（Entra管理センター）
✗ 不正解 ① 正しいユーザー同意と管理者の同意の権限が設定されていることを確認する ▼

「ユーザー同意 + 管理者同意の両方」= 委任されたアクセス許可（Delegated）のシナリオ。
今回は非対話型アプリなので Application 権限を使っており、ユーザー同意という概念が存在しない。

なぜユーザー同意が存在しないのか ユーザー同意は「サインインしたユーザーが自分の代わりにアプリを動かすことを許可する」行為。 非対話型アプリにはサインインするユーザー自体がいないため、ユーザー同意のプロセスが物理的に発生しない。
✗ 不正解 ② Microsoft Defender for Cloud でエンタープライズアプリケーションを確認する ▼

Microsoft Defender for Cloud は Azure リソースのセキュリティ状態・脆弱性・推奨事項を監視するサービス。
エンタープライズアプリケーションの権限設定を確認・管理する場所ではない。

サービス主な用途エンタープライズアプリ管理
Defender for Cloud Azureリソースのセキュリティ監視・推奨事項 ✗ 対象外
Microsoft Entra ID ID管理・アプリ登録・権限・同意の管理 ✓ ここが正解
✓ 正解 ③ 正しい管理者の同意の権限のみが設定されていることを確認する ▼

非対話型アプリ（今回のメールスキャンシステム）は Application 権限を使用。
Application 権限には管理者同意のみが必要・存在する。

✅ 「のみ」という表現がポイント ユーザー同意 + 管理者同意の両方ではなく、管理者同意のみ。
非対話型アプリに対してユーザー同意権限が設定されている場合、それは誤設定・不要な権限。 むしろセキュリティリスクになる可能性もある（最小権限の原則）。

確認場所：Entra管理センター → エンタープライズアプリケーション → 対象アプリ → 「APIのアクセス許可」→
「管理者の同意が与えられています」列が ✓ になっているか確認

✓ 正解 ④ Microsoft Entra ID でエンタープライズアプリケーションを確認する ▼

エンタープライズアプリケーションの権限設定はMicrosoft Entra ID（Entra管理センター）で管理する。
これが「どこで確認するか」という操作場所の正解。

📍 確認の具体的な手順 1. Entra管理センター（entra.microsoft.com）にアクセス
2. エンタープライズアプリケーション → すべてのアプリケーション
3. 対象アプリ（メールスキャンシステム）を選択
4. セキュリティ → 「アクセス許可」または「APIのアクセス許可」タブ
5. 「アプリケーション」権限に管理者同意が付与されているか確認
💡 正解の組み合わせまとめ

正解：③ + ④

  • ③「管理者同意のみ」= 非対話型アプリには Application 権限 + 管理者同意のみ存在する（権限種別の正確性確認）
  • ④「Entra ID で確認」= エンタープライズアプリの権限管理はEntra ID上で行う（確認場所の正確性）
🎯 パターン認識と解き方のコツ

📌 パターン①：非対話型キーワード検出

「非対話型」「バックグラウンド」「デーモン」「サービス」「自動化」「スケジュール実行」→ Application権限 + 管理者同意のみ、と即座に結びつける。

📌 パターン②：確認場所の問題

エンタープライズアプリ / アプリ登録 / 権限 / 同意 → 確認場所は Microsoft Entra ID（Entra管理センター）。Defender for CloudやAzureポータルのリソースではない。

📌 パターン③：両方 vs のみ

「ユーザー同意と管理者同意の両方」→ Delegated権限のシナリオ（対話型）
「管理者同意のみ」→ Application権限のシナリオ（非対話型）

📌 パターン④：Defender for Cloud の罠

「エンタープライズアプリを確認する」設問にDefender for Cloudが出てきたら誤り選択肢。Defender for CloudはAzureリソースのセキュリティ監視専用。

📌 パターン⑤：最小権限の原則

非対話型アプリにユーザー同意権限が設定されているのは過剰設定（誤り）。問われたら「不要な権限が付いている → 削除せよ」が正解になることがある。

📌 パターン⑥：テナント設定との関係

管理者はテナント全体でユーザー同意を禁止することもできる。「ユーザーはアプリに同意できない設定」の場合、全アプリが管理者同意必須になる。
