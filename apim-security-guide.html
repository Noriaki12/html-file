🔌API とは何か
API Application Programming Interface

アプリケーション同士が会話するための「窓口」。クライアントアプリが「〇〇のデータをください」とリクエストを送り、サーバーが「はいどうぞ」とデータを返す仕組み。

🍽️ たとえ話 レストランの注文システム。お客様（クライアントアプリ）が注文票（リクエスト）をウェイター（API）に渡し、厨房（バックエンドサーバー）が料理（データ）を作って返す。
APIは「何を注文できるか・どう注文するか」を定めたルールブック。
🚦Azure API Management とは
APIM Azure API Management

会社が提供する複数のAPIを一元管理するゲートウェイサービス。クライアントとバックエンドAPIの間に立ち、認証・流量制限・ログ・変換などを横断的に管理する。

🏢 たとえ話 大企業のセキュリティゲート（守衛室）。来訪者（クライアント）が社員証（APIキー・トークン）を提示し、目的の部署（バックエンドAPI）に案内するかどうかを判断する。
全来訪者の記録もつけて（ログ）、混雑時には入館制限（流量制御）もできる。

🛡️APIセキュリティの3つの層
💡 APIセキュリティは「誰か」「どこから」「何ができるか」の3層で考える 認証（誰か）・ネットワーク制御（どこから）・アクセス制御（何ができるか）の組み合わせが鉄則。
🔐
認証・認可（Authentication / Authorization）

「誰がアクセスしているか」を確認し「何をしてよいか」を制御する。OAuth 2.0、APIキー、サブスクリプションキーなど。

AUTH
🌐
ネットワーク制御（Network Control）

「どこからアクセスできるか」を制限する。プライベートエンドポイント、VNet統合、IPフィルタリングなど。

NETWORK
🎫

「どのAPIエンドポイントに・どれだけアクセスできるか」を管理する。サブスクリプションキー、ポリシー設定など。

ACCESS
🚫

🔐OAuth 2.0 とは
OAuth 2.0 委任認可プロトコル

「ユーザーの代わりにアプリが別サービスのリソースにアクセスする」ための標準プロトコル。
APIMでは Entra ID（旧Azure AD）などのIdPと連携し、クライアントが正規のトークンを持っているかを検証する。

🎟️ たとえ話 映画館の入場。チケット（アクセストークン）を発行会社（Entra ID）から購入し、上映時間に提示すれば入場できる。
チケットには「どの映画・どの日時・何席」という情報（スコープ）が書かれており、有効期限もある。

🌐プライベートエンドポイントとは
Private Endpoint ネットワーク隔離

APIMをインターネットから切り離し、Azure仮想ネットワーク（VNet）内のプライベートIPアドレスでのみアクセス可能にする機能。インターネットからは一切到達できなくなる。

🏢 たとえ話 会社のシステムに「社内ネットワークからしかアクセスできない専用回線」を引く行為。
公道（インターネット）からは来られず、社内の専用通路（プライベートエンドポイント）を通った人だけが入れる。
外部の攻撃者はそもそも「入り口にたどり着けない」状態になる。
✅ どんな場合に使うか 社内システム・パートナー企業など、特定のVNet内のクライアントのみがAPIを使う場合。 バックエンドへの接続にも使え、通信全体をインターネットに露出しない設計が可能。
🎫サブスクリプションキーとは
Subscription Key APIM固有のアクセス制御

Azure API Management 固有のアクセスキー。APIを使いたいクライアントは事前にサブスクリプション登録を行い、発行されたキーをリクエストヘッダーに含める。

🗝️ たとえ話 会員制サービスの会員証。事前に登録（サブスクリプション）して会員証（キー）を受け取り、入場時に提示する。
「誰がどのAPIをどれだけ使ったか」をキー単位で把握でき、不正利用の場合はキーを即座に無効化できる。
特徴内容
送信方法Ocp-Apim-Subscription-Key ヘッダーまたはクエリパラメータ
スコープ製品単位・API単位・全API で発行可能
管理Entra ポータルまたは開発者ポータルで発行・無効化
用途外部開発者へのAPI提供、クォータ管理
🚧IPホワイトリストとは（補足）
IP Filtering ネットワーク制御

APIMのポリシーで「特定のIPアドレスからのリクエストのみ許可」するフィルタリング機能。

⚠️ 有効だが「クライアントアプリ向け」では限界がある サーバー間通信・固定IPを持つシステムには有効。ただし、スマホ・PC・モバイルユーザーなど動的IPを持つ「クライアントアプリ」向けAPIには不向き。
今回の問題は「クライアントアプリ向けのAPI」なので、他の3つが優先される。
🚫アプリコードへの認証情報ハードコード — なぜ絶対NG か
🚨 選択肢②「アプリケーションコードに API 資格情報を保存する」は最悪のセキュリティ対策 これは「セキュリティ対策を実装する」どころか、セキュリティを破壊する行為。試験では必ず誤り選択肢として登場する。
なぜコードに認証情報を書いてはいけないか
1
Git リポジトリに漏れる

ソースコードをGitHubにプッシュすると、コードに書かれたAPIキー・パスワードがそのまま全世界に公開される。実際に1分以内に自動スキャナーに発見される事例が多数報告されている。

2
ログ・エラーメッセージに混入する

スタックトレースやデバッグログに認証情報が含まれて記録される可能性がある。ログ管理システムへのアクセス権があれば誰でも取得可能になる。

3
ローテーションが困難になる

キーを変更するたびにコードを修正 → テスト → デプロイのサイクルが必要になる。インシデント発生時の即時対応ができなくなる。

4
逆コンパイルで抽出される

コンパイル済みのアプリケーションでも、リバースエンジニアリングツールで文字列を抽出することが可能なケースがある。

正しい代替手段
方法概要Azure での実装
Key Vaultシークレットを安全に保存・参照Azure Key Vault + Managed Identity
環境変数実行環境から注入（コードに書かない）App Service 環境変数・K8s Secrets
Managed Identityコードに認証情報不要でAzureリソースにアクセスAzure Managed Identity
📝問題シナリオの完全解析

📋 シナリオ整理

  • クライアントアプリケーション向けのAPIを開発・提供している
  • Azure API Management（APIM）を導入してAPIを管理している
  • これらのAPIへのセキュアなアクセスを確保する必要がある
🎯 問題：どの3つのセキュリティ対策を実装すべきか？
→ 認証・ネットワーク・アクセス制御の組み合わせが正解のポイント
✓ 正解 ① OAuth 2.0 を実装する ▼

APIへのアクセスに 標準的な認証・認可プロトコル（OAuth 2.0） を使うことで、クライアントの身元確認と権限の範囲制御が可能になる。

✅ 正解な理由 APIMはEntra IDなどのOAuth 2.0対応IdPと連携し、JWTトークン検証ポリシーを設定できる。
クライアントアプリ向けAPIのセキュリティには最も標準的かつ強固な認証手段。
スコープによる細かい権限制御も可能で、セキュリティのベストプラクティスに合致する。
✗ 不正解 ② アプリケーションコードに API 資格情報を保存する ▼

これはセキュリティ対策ではなくセキュリティ脅威。OWASP Top 10でも「機密情報の不適切な管理」として上位に挙げられるアンチパターン。

🚨 完全な誤り コードへのハードコードはGit漏洩・ログ混入・逆コンパイルリスクを生む。
正解：Azure Key VaultとManaged Identityを使い、コードに認証情報を持たせない設計にする。
✓ 正解 ③ プライベートエンドポイントを使用する ▼

APIMをVNet内のプライベートIPのみでアクセス可能にし、インターネットからの直接アクセスを遮断するネットワークレベルの保護。

✅ 正解な理由 認証を突破される前にそもそも「入り口にたどり着けない」状態を作れる。
認証レイヤー（OAuth/サブスクリプションキー）と組み合わせることで多層防御（Defense in Depth）が実現する。
DDoS攻撃・スキャン攻撃のリスクも大幅に低減できる。
✗ 不正解 ④ IP ホワイトリストを定義する ▼

IP フィルタリングはAPIMで設定可能な機能だが、「クライアントアプリ向け」APIには適さない。

⚠️ なぜ不正解か クライアントアプリのユーザーは動的IP（スマホのLTE・自宅Wi-Fiなど）を持つため、全員のIPを事前に登録することは不可能。
IPホワイトリストが有効なのは、固定IPを持つサーバー間通信やB2B連携など。
今回の問題文には「クライアントアプリケーション向け」と明記されているため不適切。
✓ 正解 ⑤ サブスクリプションキーを使用する ▼

APIM固有のアクセス制御機能。クライアントは事前登録で発行されたキーをリクエストに含める。

✅ 正解な理由 APIMの基本的なアクセス制御メカニズム。クライアントごとにキーを発行してアクセスを管理し、
不正利用時は即座に特定のキーだけ無効化できる。使用量・クォータ管理にも使える。
OAuth 2.0と組み合わせることで多重のアクセス制御が可能。
💡正解まとめ

正解：① + ③ + ⑤

  • ① OAuth 2.0 → 認証・認可（誰がアクセスするかを制御）
  • ③ プライベートエンドポイント → ネットワーク保護（どこからアクセスできるかを制御）
  • ⑤ サブスクリプションキー → アクセス制御（APIM固有のキー管理）
選択肢判定セキュリティ層理由
① OAuth 2.0✓ 正解認証・認可標準的・強固・スコープ制御可能
② コードに保存✗ 不正解—セキュリティの破壊。絶対NG
③ プライベートエンドポイント✓ 正解ネットワークインターネット露出ゼロ・多層防御
④ IPホワイトリスト✗ 不正解ネットワーククライアントアプリの動的IPに不対応
⑤ サブスクリプションキー✓ 正解アクセス制御APIM固有・クライアント管理・即時無効化可能
🎯類似問題を解くためのパターン認識

📌 パターン①：コード保存は常に誤り

「アプリコードに認証情報を保存」「ハードコード」「設定ファイルに直書き」→ 常に誤り選択肢。APIキー・パスワード・証明書は全てKey Vaultへ。

📌 パターン②：対象に応じてネットワーク制御を選ぶ

動的IP（クライアントアプリ・モバイル）→ IPホワイトリストは不適切
固定IP（サーバー間通信・B2B）→ IPホワイトリストが有効
VNet内のみ → プライベートエンドポイント

📌 パターン③：APIMの3大セキュリティ機能

APIM試験では以下がセットで出る：
① OAuth 2.0（認証）
② プライベートエンドポイント（ネットワーク）
③ サブスクリプションキー（アクセス制御）

📌 パターン④：多層防御の考え方

「一つのセキュリティ対策で全部守る」は正解にならない。認証+ネットワーク+アクセス制御の組み合わせ（Defense in Depth）が選択肢に並ぶ。

📌 パターン⑤：OAuth vs サブスクリプションキーの違い

OAuth = ユーザー/アプリの身元確認（外部IdPと連携）
サブスクリプションキー = APIMへのアクセス権（APIM内部の管理）
両方同時に設定することが多い。

📌 パターン⑥：Runbook との混同に注意

APIMセキュリティの問題でRunbookが出てきたら誤り。RunbookはAzure Automationの機能でAPIセキュリティとは無関係。
