🔄Update Management とは何か
Update Management 定義

Azure の VM（Windows / Linux）に対するOS・ソフトウェアのパッチ適用を、スケジュール管理・状態把握・自動化できるサービス。
大量のVMを手作業でパッチ当てするのは非現実的 → これを自動化・可視化するのが目的。

🏥 わかりやすいたとえ 病院の全患者（VM）を定期健診（パッチ適用）させる仕組み。
誰に何のワクチン（パッチ）が必要か把握して（Log Analytics）、 スケジュールを組んで（Azure Automation）、 実際に接種する（エージェント経由でパッチ適用）。

「何が必要か把握する」「いつやるか決める」「実際にやる」の3役割を3サービスが担う。 Update Management はこの3つを組み合わせたソリューション（サービス単体ではない）。

🗄️Log Analytics Workspace とは
Log Analytics Workspace 概要

Azureのログ・メトリクスを収集・保存・クエリするためのデータストア。
VMやサービスから送られてくる「ログデータの保管庫＋分析エンジン」。

📊 たとえ話 工場（VM群）の各機械から送られてくる稼働ログを集める「中央の管理室のデータベース」。
「機械Aは今日15件のエラーが出た」「機械Bのパッチが3つ未適用」をここで把握できる。
🔗 VMとの接続方法 VM に「MMA（Microsoft Monitoring Agent）」または「OMS Agent（Linux）」をインストールし、 ワークスペースIDとキーを設定する。これが「linux1 を Log Analytics workspace に接続する」という操作の実態。
⚙️Azure Automation Account とは
Azure Automation 概要

Azureリソースの操作・管理を自動化するサービス。Runbook（スクリプト）の実行、スケジュール設定、構成管理（DSC）、Update Management などを提供。

⏰ たとえ話 「毎週日曜日の深夜2時に全VMのパッチ適用を実行せよ」というスケジュールを組む予約スケジューラー。
Runbook = パッチ適用などの手順書。Automation = その手順書を決まった時間に実行する人。
重要 Automation と Workspace の関係

Update Management を使うには、Automation Account と Log Analytics Workspace をリンク（関連付け）する必要がある。

🔄Update Management が動くまでの全体フロー
セットアップから自動パッチ適用までの5ステップ
1
Log Analytics Workspace を作成する

Update Management 用のワークスペースを用意。Automation Account と同じリージョン（またはペアリージョン）に作成する必要がある。

2
Azure Automation Account を作成し Workspace とリンクする

Automation Account に Update Management ソリューションを有効化し、Step1のWorkspaceと接続する。

3
VMにエージェントをインストールし Workspace に接続する

Linux VMならOMS Agent、Windows VMならMMAをインストール。WorkspaceIDとキーを設定してデータ送信を開始。これが「VMをWorkspaceに接続する」という操作。

4
Update Management がVMの状態を把握する

VMからOSパッチ情報がWorkspaceに送られ、「何が適用済みか・何が未適用か」が可視化される。

5
スケジュールを設定してパッチを自動適用する

Automation Account 上でメンテナンスウィンドウとスケジュールを設定。定期的に自動パッチ適用が実行される。

注意 Runbook と Update Management の関係

Runbook（選択肢①）はAzure Automationで実行できる汎用スクリプト。Update Management はRunbookとは別の専用機能として提供されており、Update Managementのためにわざわざ Runbook を定義する必要はない。

⚠️ 試験頻出の罠 「Update Management = Runbook」と誤解しやすいが、Update Management はAutomation Account の中の独立したソリューション。Runbookは「自分でスクリプトを書いて実行する」機能で別物。
🗺️リージョン制約 — 今回の問題の核心
🚨 Update Management の最重要制約 Azure Automation Account と Log Analytics Workspace はサポートされた同一リージョンまたはリージョンペアでないとリンクできない。
これが守られていないと Update Management を有効化できない。
今回の問題の構成
linux1 VM East US RG: prod
✗リージョン
不一致
prod-workspace Central US RG: prod
問題点：East US の VM と Central US の Workspace はペアリングが成立しない Update Management のために Automation Account を作っても、Central US の Workspace とリンクできない。
→ 既存の prod-workspace はこのVMのUpdate Managementには使えない。

📝問題シナリオの完全解析

📋 シナリオ整理

  • linux1（Linux VM） — リソースグループ: prod、リージョン: East US
  • prod-workspace（Log Analytics）— リソースグループ: prod、リージョン: Central US
  • linux1 で Update Management ソリューションを有効化したい
🎯 問題の核心：VMはEast USなのに既存WorkspaceはCentral US。
Update Managementのリージョン制約を満たしていないため、そのまま使えない。
→ まず何をすべきか？
✗ 不正解 ① Azure Automation の runbook を定義する ▼

Runbook は Azure Automation の「自前スクリプト実行機能」。Update Management は Runbook とは別の専用ソリューション。

順序の問題もある 仮にRunbookが必要だとしても、Workspaceのリージョン問題が解決されていない状態では何も進まない。 「最初にすること」ではない。
✗ 不正解 ② linux1 を Log Analytics workspace に接続する ▼

VMをWorkspaceに接続するのは確かに必要なステップ。しかし「最初に」やることではない。

🚨 根本問題が解決していない prod-workspace は Central US にある。East US の linux1 をこの Workspace に繋いでも、
Update Management のために使う Automation Account と Workspace がリージョンペア要件を満たせない。
つまり 正しいリージョンの Workspace がまず必要。接続はその後。
✗ 不正解 ③ linux1 で boot diagnostics を有効化する ▼

boot diagnostics はVMの起動画面・ログを取得する診断機能。VMが起動しない時のトラブルシューティングに使う。

🚨 Update Management とは完全に無関係 パッチ管理の前提条件でも何でもない。試験でよく登場する「もっともらしい罠選択肢」の典型。
✓ 正解 ④ 2つ目の Log Analytics workspace をデプロイする ▼

既存の prod-workspace（Central US）は East US のlinux1 向け Update Management には使えない。East US（またはペアリージョン）に新しい Workspace を作成する必要がある。

✅ これが「最初にすること」な理由 正しいリージョンのWorkspaceがなければ、それ以降のステップ（Automation Account作成・VMの接続）が全部成立しない。
土台となるWorkspaceを正しいリージョンに用意することが、依存関係の起点になる。

その後の順序：
④ East USに新Workspaceデプロイ
→ Automation Account（East US）を作成してWorkspaceとリンク
→ Update Management ソリューションを有効化
→ linux1にエージェントをインストールして新Workspaceに接続
→ パッチスケジュール設定

💡正解の根拠まとめ

なぜ④が「最初にすること」なのか

  • Update Management の前提：Automation Account と Workspace が同一リージョン（またはペア）でなければならない
  • 既存 prod-workspace は Central US → East US の VM 向けの Automation Account とリンク不可
  • よって、East US に対応した新しい Workspace がなければ何も始まらない
  • 「最初に」= 他の全ステップの依存関係の起点になる操作
🎯類似問題を解くための思考フレーム

📌 パターン①：リージョン不一致を検出する

VMのリージョンとWorkspaceのリージョンが異なる → まず「正しいリージョンにWorkspaceを作れ」が正解になりやすい。問題文に2つのリージョンが出てきたら要注意。

📌 パターン②：「最初に」という問い方

「最初に何をすべきか」= 依存関係の根本を問う問題。他の全ステップが依存する前提条件（インフラの土台）を選ぶ。ここではWorkspaceが土台。

📌 パターン③：Runbook の誤誘導

「Automationが出てきた → Runbookが必要」は罠。Update ManagementはRunbookとは独立した専用ソリューション。Runbookは「自前スクリプト」の話。

📌 パターン④：boot diagnostics の罠

boot diagnosticsはVM起動診断機能。Update Management・パッチ管理とは無関係。「もっともらしい」だけの罠選択肢として頻出。

📌 パターン⑤：VMの接続は「最初」ではない

「VMをWorkspaceに接続する」は確かに必要なステップだが、正しいWorkspaceが存在しない状態ではできない。土台が整ってから繋ぐ。

📌 パターン⑥：Automation = Workspace とセット

Azure Automation Account は単体では機能しない。Update Management には必ずLog Analytics Workspaceとのリンクが必要。どちらが欠けてもセットアップ不可。
