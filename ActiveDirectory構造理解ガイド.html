01

基本概念：ADとは何を管理するものか

💡
一言で言うと

Active Directory（AD）は「誰が・どのリソースに・どんな権限でアクセスできるか」を一元管理するWindowsのオンプレID基盤。

オブジェクト説明例
ユーザーアカウント人間が使うID。認証情報・属性情報を持つyamada@contoso.com
コンピューターアカウントドメイン参加したPC・サーバーPC-YAMADA$
グループユーザーをまとめ権限管理の単位とするSales_Group
OU（組織単位）GPO適用・管理委任の入れ物東京本社OU
GPOポリシーを一括強制するオブジェクトパスワード最低長8文字
サービスアカウントアプリ・サービスが使うID。人間は不使用svc_sql

🏠 ドメイン（Domain）

管理・認証・ポリシーの基本単位。DNSベースの名前（contoso.com）を持ち、DCがそのドメインのIDを管理する。

🌳 ツリー（Tree）

連続するDNS名前空間を持つドメインの集合。親子ドメイン間は自動で双方向トラストが張られる。

🌲 フォレスト（Forest）

ADの最上位管理単位。スキーマを共有。セキュリティ境界はフォレスト単位（ドメインではない）。

📂 OU（組織単位）

ドメイン内のオブジェクトを整理する論理的な入れ物。GPO適用・管理委任の主な単位。ドメインを超えない。

⚠️
よくある誤解：ドメインはセキュリティ境界ではない

同一フォレスト内ではDomain Adminsがフォレスト全体を掌握できる。隔離したいなら別フォレストにする必要がある。

種別方向推移性設定主な用途

フォレストトラスト双方向/片方向推移的手動異なるフォレスト間。M&A・会社間連携。Selective Auth推奨。
外部トラスト双方向/片方向非推移的手動特定ドメインのみを繋ぐ。影響範囲が限定的。レガシー環境との接続。
ショートカットトラスト双方向/片方向推移的手動大規模フォレストで認証パスを短縮するパフォーマンス最適化目的。
🚨
セキュリティ観点：信頼関係は攻撃経路になる
  • フォレストトラストがあると、一方が侵害されたときトラスト経由で横展開される可能性がある
  • SIDフィルタリング（SID Filtering）の設定状況を必ず確認する
  • フォレストトラスト設定時は Selective Authentication を有効化するのがベストプラクティス
04

認証の仕組み：Kerberos と NTLM

Kerberosフロー（正常系）
1
AS-REQ（認証要求）Client → KDC
ユーザーがログイン。クライアントがKDC（DC上のKerberos認証サービス）にTGT発行を要求する。
2
AS-REP（TGT発行）KDC → Client
KDCがTGT（Ticket Granting Ticket）を発行。このチケットでリソースへのチケット要求が可能になる。
3
TGS-REQ（サービスチケット要求）Client → KDC
TGTを使って特定リソース（ファイルサーバー等）へのサービスチケットをKDCに要求する。
4
TGS-REP（サービスチケット発行）KDC → Client
KDCがサービスチケット（ST）を発行する。
5
リソースアクセスClient → Server
STを提示してリソースにアクセス。パスワードはネットワークに流れない。
NTLMフロー（フォールバック）
1
NEGOTIATEClient → Server
クライアントがNTLM認証の開始を要求する。
2
CHALLENGE（乱数）Server → Client
サーバーがランダムな8バイトの値（Challenge）を送る。
3
AUTHENTICATE（ハッシュ応答）Client → Server
クライアントがNTLMハッシュとChallengeを組み合わせたResponseを返す。
4
DC検証Server → DC
サーバーがDCにResponseを転送して検証させる。
⚠️
NTLMのリスク

NTLMハッシュが盗まれるとPass-the-Hash攻撃が可能。Kerberosが使えない場合（IP直打ちアクセス等）のみ使われる。可能な限り制限・無効化を推奨。

05

主要コンポーネント

🖥️ ドメインコントローラー（DC）

ADデータベース（ntds.dit）を保持。認証・ディレクトリ検索・レプリケーションを担う。通常2台以上で冗長化。

  • Kerberos KDCの役割も兼務
  • FSMOロールを持つDCが存在する
  • 拠点があればサイトごとに配置が望ましい
📚 グローバルカタログ（GC）

フォレスト全体のオブジェクトの部分レプリカを持つDC。フォレスト横断の検索・ユニバーサルグループ解決に使われる。

  • ポート：3268（GC）/ 3269（GC over SSL）
  • 各サイトに1台以上のGCを配置するのが推奨
  • GCダウン＝ログイン遅延の原因になる
FSMOロール（5つの単一マスタ操作）
プロトコル / ポート用途注意点
Kerberos（88/TCP,UDP）主要認証プロトコル。チケットベースADではデフォルト。NTLMフォールバックに注意
LDAP（389/TCP）ディレクトリ検索・オブジェクト操作平文。LDAPS（636）または署名を強制すること
LDAPS（636/TCP）LDAP over TLS証明書が必要
DNS（53）DCロケート・ADの根幹AD統合DNSゾーンが一般的
SMB（445/TCP）SYSVOL共有・GPO配布DCとクライアント間で必須
GC（3268/3269）グローバルカタログ検索フォレスト横断の検索・UG解決に必須
RPC（135 + 動的）レプリケーション・管理ツールFW越えは動的ポートに注意

セキュリティ観点での整理

典型的な攻撃パス（AD侵害の流れ）
🎣
初期侵入 — フィッシング・VPN脆弱性・公開サーバー侵害で一般ユーザーのセッション/認証情報を取得
Tier 2
↓
🔄
横展開（Lateral Movement） — Pass-the-Hash / Pass-the-Ticket でネットワーク内を移動。Kerberoastingで特権アカウントのハッシュを取得
Tier 1
↓
👑
特権昇格（Privilege Escalation） — Domain Admins / Enterprise Admins 相当の権限を奪取
Tier 0
↓
💀
DCSync / Golden Ticket — krbtgtハッシュ取得。フォレスト全体の永続的掌握。ランサムウェア展開・データ窃取
Tier 0
🎯 Tier 0 資産（最重要保護対象）
  • すべてのドメインコントローラー
  • フォレストルートDCのすべて
  • Enterprise Admins / Schema Admins メンバー
  • Entra Connect（AD Connect）サーバー
  • AD FSサーバー（フェデレーション構成の場合）
  • krbtgtアカウント
⚡ 代表的な攻撃手法
  • Pass-the-Hash：NTLMハッシュを使った横展開
  • Pass-the-Ticket：Kerberosチケットの再利用
  • Kerberoasting：SPNを持つSAの平文パス解析
  • DCSync：レプリケーション権限悪用でハッシュ全件取得
  • Golden Ticket：krbtgtハッシュで偽チケット作成
  • AiTM：フィッシング＋中間者でMFAバイパス

顧客ヒアリングチェックリスト

✅
使い方

各項目をタップ/クリックで確認済みマーク。ヒアリング前の準備・実施中のチェックとして活用。

📐 基本構成
✓
フォレストはいくつありますか？
M&A歴のある企業は複数フォレストの可能性が高い。1か複数かで信頼関係の複雑さが変わる。
✓
ドメインはいくつ、どんな名前ですか？（DNS名 / NetBIOS名）
子ドメイン・別ツリーの有無を確認。構成図があれば共有してもらう。
✓
DCは何台、どこに置いてありますか？（拠点・クラウド）
クラウドDC（Azure上）かオンプレのみか。拠点が多い場合はサイト設計も確認。
✓
ADのユーザー数（オブジェクト数）はだいたい何件ですか？
ライセンス見積もり・エージェント展開計画・SIEMインジェスト量の前提になる。
✓
Windowsサーバー / クライアントのバージョン構成は？
Win Server 2008以前はKerberos armoring不対応など制約が多い。
🔗 信頼関係・連携
✓
フォレスト間・ドメイン間の信頼関係はありますか？
どの方向のトラストか。双方向か片方向か。
✓
フォレストトラストがある場合、Selective Authenticationは有効ですか？
無効だとトラスト先の全ユーザーがデフォルトでリソースアクセス可能になる。
✓
Entra ID（旧Azure AD）と連携していますか？
Entra Connect / Cloud Sync / AD FSのどれか。またはEntraなしのオンプレのみか。
✓
Entra Connect使用の場合、同期方式は？（PHS / PTA / AD FS）
PHSが最もシンプルでMFAバイパス攻撃への耐性が高い。PTAはDC依存。
✓
外部パートナーとのADトラストはありますか？
B2B連携。パートナー側の侵害が自社に影響しうるか確認。
🛡️ セキュリティ・権限管理
✓
Domain Admins / Enterprise Admins のメンバー数と管理方法は？
多すぎる場合は特権管理の問題あり。PIM（Privileged Identity Management）の有無も確認。
✓
ADの監査ログをどこに保管・監視していますか？
SIEMに転送しているか（Sentinel等）。何日保管か。
✓
Defender for Identity（旧Azure ATP）は導入していますか？
DCへのセンサー展開が前提。どのDCに入れているか。
✓
NTLMを無効化・制限していますか？
完全無効は困難だが制限の有無を確認。Pass-the-Hash対策の基本。
✓
LDAP署名・LDAPS（LDAP over TLS）を強制していますか？
LDAP平文はMitM攻撃リスクあり。2020年3月MSパッチ以降は要対応。
✓
Entra Connect サーバーのセキュリティ管理状況は？
Tier 0扱いが必要。オンプレとEntra両方への書き込み権限を持つ。
🏢 組織・運用体制
✓
ADの運用はIT部門の何名が担当していますか？
小規模チームは自動化・アラート対応の余力が少ない。MSSP検討の材料にもなる。
✓
M&A・組織変更の予定はありますか？
フォレスト統合・分離・新ドメイン追加が近く起きる可能性を確認。
✓
Entra ID（クラウドID）への段階的移行計画はありますか？
Entra参加デバイスへの移行・AD FS廃止・パスワードレス推進など。
✓
以前にAD関連のセキュリティインシデントはありましたか？
過去の侵害経験は現状の課題認識と優先度を把握する上で重要。
✓
ADアカウントの棚卸し（退職者アカウント整理）はいつ実施しましたか？
退職者アカウントが残存しているケースは非常に多い。基本的な課題の有無がわかる。
09

よくある顧客構成パターン

パターン A シングルドメイン・シングルフォレスト

SMB〜中堅企業に最多。DC2台冗長化 + Entra Connect でM365連携するパターンが主流。

パターン B 親子ドメイン・単一フォレスト

地域・法人ごとに管理を分けたい大企業。Domain Admins権限がフォレスト全域に及ぶことを意識する。

パターン C M&A後・複数フォレスト（統合前）

M&A直後に多い。フォレストが分離しているためセキュリティ境界は守られるが、連携にフォレストトラスト設定が必要。

パターン D ハイブリッドID（現在の主流）

多くの日本企業が該当。PHS + 条件付きアクセスが現在のベストプラクティス。段階的にクラウドIDへ移行中。
