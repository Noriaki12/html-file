AZ-104 / SC-300 試験対策

Azure RBAC と
Microsoft Entra ID ロールの完全攻略

権限・スコープ・継承の仕組みを理解して類似問題を確実に解く

01 最重要前提 — 2つのロール体系は完全に独立

🔑 試験の核心：Entra IDロール ≠ Azure RBACロール Microsoft製品には「テナント（Entra ID）」と「Azureサブスクリプション」という2つの独立した管理面がある。 どちらのロールを持っていても、もう一方には自動的に権限が付与されない。
🪪 Microsoft Entra ID ロール

テナント操作に効く

  • ユーザー・グループの作成・削除
  • アプリ登録・Enterprise Apps管理
  • Conditional Accessポリシー管理
  • MFA/SSPR設定
  • Entra ID自体の設定変更
Azure リソース（VM・ストレージ等）へのアクセスは付与しない
🛡️ Azure RBAC ロール

Azureリソース操作に効く

  • VM・ネットワーク・ストレージの操作
  • リソースグループ内のリソース管理
  • RBAC ロール割り当て（Ownerのみ）
  • コスト・請求情報の参照
  • サブスクリプション設定の変更
Entra ID テナント操作（アプリ登録など）は行えない
⚠️ 唯一の例外：Global Administrator Entra IDの「全体管理者」は、Azure上のサブスクリプション管理権限を昇格操作で取得できるが、デフォルトでは持っていない。試験では「GA = Azure全権限あり」と思い込まないこと。

02 スコープと継承の仕組み

Azure RBACはスコープ階層を持ち、上位スコープで割り当てたロールは下位に自動継承される。

管理グループ （Management Group）
↓ 継承
サブスクリプション ← User1 = Owner / User2 = Contributor / User3 = Security Admin
↓ 継承
リソースグループ resourcegroup1 ← サブスクリプションのロールがここにも適用される
↓ 継承
個別リソース （VM, Storage, etc.）
💡 継承の実際例 User1 が「サブスクリプション Owner」を持つ = resourcegroup1 でも Owner として扱われる。 リソースグループに明示的にロール設定がなくても、上位から継承されるため権限は有効。

04 問題を解くデシジョンツリー

Q1
操作の対象はどこか？
Entra/テナント アプリ登録・CA・ユーザー管理 → Entraロールを持つユーザーを探す
Azureリソース VM・RG・ストレージ等 → Azure RBACロールを持つユーザーを探す
Q2
操作の種類は何か？
権限管理 RBAC割り当て = Owner または User Access Administrator が必要
リソース作成 VM等の作成 = Owner または Contributor で可
読み取り 参照のみ = Reader 以上で可（ほぼ全員）
Q3
スコープを確認する
サブスクリプション → リソースグループにも継承される
RGスコープ限定 → そのRG内のみ有効（サブスクリプション全体には影響しない）
Q4
Azure RBACロールが「None」のユーザーは？
答え Entraロールを持っていてもAzureリソース操作は一切できない（User4 パターン）

05 元の問題 — 解答と解説

ユーザー Entra IDロール サブスクリプションロール
User1 Application Developer Owner
User2 — Contributor
User3 — Security Admin
User4 Conditional Access Administrator —
Q1. resourcegroup1 のパーミッション（RBAC）を管理できるユーザーは？
User1 のみ
RBAC ロール割り当てには Microsoft.Authorization/*/write が必要 = Owner または User Access Administrator のみ。
Contributor（User2）はリソース操作はできるが権限付与はできない。
Security Admin（User3）・Entra専用ロール（User4）も対象外。
画面の選択: User1 and User2 → 正解: User1 only
Q2. resourcegroup1 で VM を作成できるユーザーは？
User1 and User2
VM 作成には Microsoft.Compute/virtualMachines/write が必要。
Owner（User1）と Contributor（User2）は両方ともこの権限を含む。
Security Admin（User3）は Compute 権限を持たない。User4 は Azure RBAC ロール自体がない。
正解: User1 and User2 ✓
Q3. テナントにアプリを登録できるユーザーは？
User1 のみ
アプリ登録は Entra IDの操作。
User1 は Entra ID「Application Developer」ロールを保有 → 登録可。
User2・User3 は Entra ID ロールなし。User4 の「Conditional Access Administrator」はCA管理専用でアプリ登録権限なし。
※テナント設定で「全ユーザーにアプリ登録を許可」が ON の場合は全員可だが、試験では特に言及がない限りデフォルト OFF（制限あり）として判断する。
正解: User1 ✓

06 類似問題で使える解き方のコツ

TIP 1 「None」が答えを絞る最短ルート

Azure RBAC が None のユーザーは Azure 操作一切不可。Entra ロールが None のユーザーは Entra 操作一切不可。まず None を確認して除外すると残りの選択肢が絞れる。

TIP 2 RBAC管理 = Owner か UAA だけ

「ロールを割り当てる」「パーミッションを管理する」= Microsoft.Authorization/*/write。これを持つのは Owner と User Access Administrator のみ。Contributor でも不可という点は頻出。

TIP 3 スコープは上位から下位に流れる

サブスクリプション Owner → RG Owner → リソース Owner が自動的に継承される。「subscriptionA の Contributor が resourcegroup1 でリソース作成できるか」→ YES（継承されるため）。

TIP 4 操作の「場所」で種別を判断

「Azureポータルで VM/ストレージ/ネットワーク操作」→ Azure RBAC。
「Entra ポータルでアプリ/ユーザー/CA 操作」→ Entra ロール。
どちらの画面の話かを意識すると分類が速くなる。

TIP 5 Security Admin は「Compute なし」と覚える

Security Admin は Defender for Cloud や Sentinel のポリシー・アラート管理に特化。VM 作成・削除などのコンピューティング操作はできない。「セキュリティ」という名称に惑わされない。

TIP 6 アプリ登録のテナント設定に注意

デフォルトでは全ユーザーがアプリ登録可能。ただし試験問題で「Entra ロールを持つ人が答えになっている」場合は設定が制限済みと読む。問題文に「Users can register applications = No」と書かれていたら Application Developer/Admin が必要。
